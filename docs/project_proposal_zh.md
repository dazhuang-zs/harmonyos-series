# 项目立项书：HarmonyOS 开发助手（MiMo 驱动）

## 一、项目背景

### 1.1 行业现状

鸿蒙操作系统（HarmonyOS）自 2019 年发布以来，生态快速扩张。截至 2026 年 5 月：
- 鸿蒙 6 终端设备突破 **6000 万台**（华为官方数据）
- 注册开发者超过 **1000 万**（华为终端 BG CEO 何刚公布）
- 日均新增用户 **数十万**，预计今冬突破 1 亿大关

然而，鸿蒙开发存在明显痛点：
1. **学习曲线陡峭**：ArkTS 语法与 Android/Java 差异大，文档分散
2. **报错诊断困难**：编译错误提示不够友好，新手难以定位问题
3. **代码迁移成本高**：Android 项目迁到鸿蒙，需要大量手动重写
4. **优质教程稀缺**：相比 Android/iOS，鸿蒙实战教程严重不足

### 1.2 机会窗口

2026 年 4 月 28 日，小米启动「百万亿 Token 创造者激励计划」，免费发放 100 万亿 Token 权益，截止 5 月 28 日。

本项目计划申请该计划，**利用 MiMo API 驱动鸿蒙开发助手**，解决上述痛点，同时积累真实项目案例，反哺 CSDN 技术文章创作。

## 二、项目目标

### 2.1 核心目标

构建一个 **MiMo 大模型驱动的鸿蒙开发助手**，提供以下能力：

| 功能模块 | 描述 | MiMo 使用场景 |
|---------|------|---------------|
| RAG 文档问答 | 基于华为官方文档 + CSDN 优质文章，回答鸿蒙开发问题 | 语义理解 + 答案生成 |
| ArkTS 代码生成 | 根据自然语言描述，生成可运行的 ArkTS/ArkUI 代码 | Code Generation |
| 报错诊断 | 粘贴编译错误，自动定位原因并给出修复方案 | Reasoning + Code Analysis |
| Android → ArkTS 迁移 | 将 Android Java/Kotlin 代码转换为等价 ArkTS 实现 | Code Translation |
| CSDN 文章自动发布 | 将问答对、代码案例整理成技术文章，自动发布到 CSDN | Content Generation |

### 3.2 LLM 抽象层设计（新增）

为避免单一 LLM 供应商锁定，设计统一抽象层：

```python
# 核心接口示意
class LLMProvider(ABC):
    async def chat(self, messages: list[dict], **kwargs) -> str: ...
    async def chat_stream(self, messages: list[dict], **kwargs) -> AsyncIterator[str]: ...

class EmbeddingProvider(ABC):
    async def embed(self, texts: list[str]) -> list[list[float]]: ...

# 实现
class MiMoProvider(LLMProvider): ...      # 主力
class DeepSeekProvider(LLMProvider): ...   # 备用（OpenAI 兼容接口）
class BGEEmbedding(EmbeddingProvider): ... # 本地部署，C-MTEB 中文榜靠前
```

**切换策略**：通过 `.env` 配置 `LLM_PROVIDER=mimo` / `deepseek` / `openai`，一行切换。

### 3.3 技术栈

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| 后端 | Python 3.11 + FastAPI | 快速开发，生态丰富 |
| LLM | MiMo API（主），LLM 抽象层支持 DeepSeek/OpenAI 兼容接口降级 | 核心驱动力，避免单点依赖 |
| Embedding | BGE-Large-ZH（本地部署） | 中文效果优秀的开源模型，本地部署零成本，LangChain 原生支持 |
| RAG | LangChain + Milvus | 工业级向量检索 |
| 前端 | Next.js 14 + TypeScript（M5 阶段） | 现代 Web 开发体验 |
| 数据爬取 | Playwright（SPA 渲染）+ BeautifulSoup + Scrapy | 华为文档站为 SPA，需无头浏览器；OpenHarmony 用 Git + Markdown 解析 |
| CSDN 发布 | CSDN Markdown API | 自动发布技术文章 |
| 部署 | Docker + K8s | 可扩展部署 |

#### 技术挑战：华为文档站 SPA 渲染

华为开发者文档站是 **SPA（单页应用）**，JavaScript 动态加载内容，普通的 HTTP 请求 + BeautifulSoup 无法获取正文。

**解决方案**：
```
URL 列表 → Playwright 启动无头浏览器 → 等待页面渲染完成
→ 提取 DOM → BeautifulSoup 解析正文 → 转为 Markdown
→ 按标题层级分块 → BGE-Large-ZH 向量化 → 存入 Milvus
```

#### 分块策略（Chunking）

| 策略 | 说明 |
|------|------|
| 按标题切分 | 以 h2/h3 为边界，每个 chunk 包含完整的代码示例和说明 |
| 保留上下文 | 每个 chunk 附带父级标题链（如 `组件 > Button > 属性`） |
| 代码块完整 | 不拆分代码示例，保证可读性 |
| chunk 大小 | 目标 500-1000 token，过长则二次切分，过短则合并 |

#### M1 阶段数据计划

1. **手动整理 50-100 篇核心文档 URL**（ArkTS 语言指南、常用组件、数据管理、网络请求等高频主题）
2. **编写 Playwright 爬虫脚本**，输出结构化 Markdown
3. **构建初始知识库**，跑通 RAG 流程验证效果
4. **后续迭代**：扩展到自动化爬取 + 增量更新 + 社区内容

## 四、MiMo API 使用场景（重点）

> 以下内容将直接用于小米 MiMo 激励计划申请表单填写。

### 4.1 场景 1：RAG 文档问答

**输入**：用户问题（如"ArkTS 中 @State 和 @Prop 有什么区别？"）

**处理流程**：
1. 用 BGE-Large-ZH（本地部署）将问题向量化
2. 在 Milvus 中检索 Top-5 相关文档片段
3. 将问题 + 文档片段拼成 Prompt，调用 MiMo Chat API
4. 流式返回答案

> **Embedding 选型说明**：MiMo 不提供 Embedding 接口，使用 BGE-Large-ZH 本地部署。该模型在 C-MTEB 中文榜长期排名靠前，1.3GB 模型大小可在 2 核 4G 服务器上运行，LangChain 原生支持集成。

**Token 消耗**：每次问答约 **2000-5000 Token**

### 4.2 场景 2：ArkTS 代码生成

**输入**：自然语言需求（如"写一个鸿蒙待办清单 App，支持添加、删除、本地存储"）

**处理流程**：
1. 用 MiMo 生成代码框架（.ets 文件）
2. 调用 MiMo 生成配套 XML 布局（如需要）
3. 用 MiMo 生成单元测试代码
4. 返回完整可运行的 DevEco Studio 项目

**Token 消耗**：每次生成约 **5000-10000 Token**

### 4.3 场景 3：报错诊断

**输入**：编译错误信息（如"Cannot find module '@ohos/net'..."）

**处理流程**：
1. 用 MiMo 分析错误类型和可能原因
2. 检索知识库中的相似案例
3. 生成分步修复方案（含代码示例）
4. 返回 Markdown 格式的诊断报告

**Token 消耗**：每次诊断约 **3000-6000 Token**

### 4.4 场景 4：Android → ArkTS 代码迁移

**输入**：Android Java/Kotlin 代码

**处理流程**：
1. 用 MiMo 理解原始代码逻辑
2. 生成等价的 ArkTS 实现
3. 标注迁移注意事项（如 API 差异）
4. 返回对比代码（Before/After）

**Token 消耗**：每 100 行代码约 **8000-15000 Token**

### 4.5 场景 5：CSDN 文章自动生成

**输入**：问答对 + 代码案例

**处理流程**：
1. 用 MiMo 将技术内容整理成文章大纲
2. 生成完整 Markdown 文章（含代码块、配图提示词）
3. 调用 CSDN API 自动发布到草稿箱
4. 返回文章链接

**Token 消耗**：每篇文章约 **4000-8000 Token**

## 六、风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| MiMo Token 申请失败 | 中 | 高 | LLM 抽象层已支持 DeepSeek/OpenAI 兼容接口降级 |

| 文档分块质量不佳 | 中 | 中 | 按语义标题切分而非固定字数、保留代码块完整性、人工抽检 chunk 质量 |
