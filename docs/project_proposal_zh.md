# 项目立项书：HarmonyOS 开发助手（MiMo 驱动）

## 一、项目背景

### 1.1 行业现状

鸿蒙操作系统（HarmonyOS）自 2019 年发布以来，生态快速扩张。截至 2026 年 5 月：
- 鸿蒙 6 终端设备突破 **6000 万台**（华为官方数据）
- 注册开发者超过 **1000 万**（华为终端 BG CEO 何刚公布）
- 日均新增用户 **数十万**，预计今冬突破 1 亿大关

然而，鸿蒙开发存在明显痛点：
1. **学习曲线陡峭**：ArkTS 语法与 Android/Java 差异大，文档分散
2. **报错诊断困难**：编译错误提示不够友好，新手难以定位问题
3. **代码迁移成本高**：Android 项目迁到鸿蒙，需要大量手动重写
4. **优质教程稀缺**：相比 Android/iOS，鸿蒙实战教程严重不足

### 1.2 机会窗口

2026 年 4 月 28 日，小米启动「百万亿 Token 创造者激励计划」，免费发放 100 万亿 Token 权益，截止 5 月 28 日。

本项目计划申请该计划，**利用 MiMo API 驱动鸿蒙开发助手**，解决上述痛点，同时积累真实项目案例，反哺 CSDN 技术文章创作。

## 二、项目目标

### 2.1 核心目标

构建一个 **MiMo 大模型驱动的鸿蒙开发助手**，提供以下能力：

| 功能模块 | 描述 | MiMo 使用场景 |
|---------|------|---------------|
| RAG 文档问答 | 基于华为官方文档 + CSDN 优质文章，回答鸿蒙开发问题 | 语义理解 + 答案生成 |
| ArkTS 代码生成 | 根据自然语言描述，生成可运行的 ArkTS/ArkUI 代码 | Code Generation |
| 报错诊断 | 粘贴编译错误，自动定位原因并给出修复方案 | Reasoning + Code Analysis |
| Android → ArkTS 迁移 | 将 Android Java/Kotlin 代码转换为等价 ArkTS 实现 | Code Translation |
| CSDN 文章自动发布 | 将问答对、代码案例整理成技术文章，自动发布到 CSDN | Content Generation |
| Agent 工具调用 | ReAct 模式自主决策：检索文档 → 生成代码 → 诊断错误，多步串联执行 | Planning + Tool Use |
| 多轮对话记忆 | 对话上下文窗口管理、长对话摘要压缩、用户偏好记忆 | Conversation Context Management |
| 高级 RAG（混合检索） | BM25 稀疏检索 + Dense 向量混合搜索 + Cross-encoder Rerank 重排序 + Query Rewriting 查询重写 | 检索增强 + 语义匹配 |
| 评测体系 | RAGAS 自动化评测：检索 Recall@K/MRR、生成 Faithfulness/Answer Relevance、50+ 标注测试集 | 质量评估 |
| Graph RAG | 鸿蒙 API 实体关系图谱（组件→属性→事件），Neo4j 图遍历增强检索 | 知识图谱推理 |
| 多模态识别 | 截图上传 → Vision Model 识别 UI 布局 → 生成 ArkUI 代码；编译错误截图 OCR 提取 | Vision + OCR |
| 语义缓存 | 相似问题向量匹配命中缓存（相似度 > 0.95），减少重复 LLM 调用，降低成本 | 成本优化 |
| Guardrails 安全防护 | Prompt Injection 防御、输出代码安全校验（防恶意代码）、内容合规过滤 | 安全检测 |
| 可观测性 | LangFuse 链路追踪：每次调用延迟/Token/快照、Dashboard 监控、异常告警 | 运维监控 |
| CI/CD 实验体系 | Prompt 版本管理、A/B 测试框架、自动化评测流水线（提交→评测→报告） | 工程化 |
| 模型微调 | 收集用户反馈数据构建 SFT 数据集，LLaMA-Factory LoRA 微调开源模型 | 模型定制 |

### 3.2 LLM 抽象层设计（新增）

为避免单一 LLM 供应商锁定，设计统一抽象层：

```python
# 核心接口示意
class LLMProvider(ABC):
    async def chat(self, messages: list[dict], **kwargs) -> str: ...
    async def chat_stream(self, messages: list[dict], **kwargs) -> AsyncIterator[str]: ...

class EmbeddingProvider(ABC):
    async def embed(self, texts: list[str]) -> list[list[float]]: ...

# 实现
class MiMoProvider(LLMProvider): ...      # 主力
class DeepSeekProvider(LLMProvider): ...   # 备用（OpenAI 兼容接口）
class BGEEmbedding(EmbeddingProvider): ... # 本地部署，C-MTEB 中文榜靠前
```

**切换策略**：通过 `.env` 配置 `LLM_PROVIDER=mimo` / `deepseek` / `openai`，一行切换。

### 3.4 数据获取策略（新增）

RAG 系统的质量取决于知识库的质量。本项目需要从多个来源获取鸿蒙开发文档，构建专业级知识库。

#### 数据源清单

| 来源 | URL | 获取方式 | 内容类型 | 优先级 |
|------|-----|---------|---------|--------|
| 华为开发者文档 | developer.huawei.com/consumer/cn/doc/ | Playwright 爬取 | ArkTS 指南、API 参考、开发指南 | P0 |
| OpenHarmony 仓库 | gitee.com/openharmony | Git Clone + Markdown 解析 | 源码文档、设计文档 | P1 |
| CSDN 鸿蒙专栏 | blog.csdn.net | CSDN API / 爬虫 | 实战教程、踩坑经验 | P2 |
| 华为开发者论坛 | developer.huawei.com/consumer/cn/forum/ | 爬虫 | Q&A 问答对 | P2 |
| 掘金 HarmonyOS 标签 | juejin.cn | 爬虫 | 技术文章 | P3 |

#### 技术挑战：华为文档站 SPA 渲染

华为开发者文档站是 **SPA（单页应用）**，JavaScript 动态加载内容，普通的 HTTP 请求 + BeautifulSoup 无法获取正文。

**解决方案**：
```
URL 列表 → Playwright 启动无头浏览器 → 等待页面渲染完成
→ 提取 DOM → BeautifulSoup 解析正文 → 转为 Markdown
→ 按标题层级分块 → BGE-Large-ZH 向量化 → 存入 Milvus
```

#### 分块策略（Chunking）

| 策略 | 说明 |
|------|------|
| 按标题切分 | 以 h2/h3 为边界，每个 chunk 包含完整的代码示例和说明 |
| 保留上下文 | 每个 chunk 附带父级标题链（如 `组件 > Button > 属性`） |
| 代码块完整 | 不拆分代码示例，保证可读性 |
| chunk 大小 | 目标 500-1000 token，过长则二次切分，过短则合并 |

#### M1 阶段数据计划

1. **手动整理 50-100 篇核心文档 URL**（ArkTS 语言指南、常用组件、数据管理、网络请求等高频主题）
2. **编写 Playwright 爬虫脚本**，输出结构化 Markdown
3. **构建初始知识库**，跑通 RAG 流程验证效果
4. **后续迭代**：扩展到自动化爬取 + 增量更新 + 社区内容

## 四、MiMo API 使用场景（重点）

> 以下内容将直接用于小米 MiMo 激励计划申请表单填写。

### 4.1 场景 1：RAG 文档问答

**输入**：用户问题（如"ArkTS 中 @State 和 @Prop 有什么区别？"）

**处理流程**：
1. 用 BGE-Large-ZH（本地部署）将问题向量化
2. 在 Milvus 中检索 Top-5 相关文档片段
3. 将问题 + 文档片段拼成 Prompt，调用 MiMo Chat API
4. 流式返回答案

> **Embedding 选型说明**：MiMo 不提供 Embedding 接口，使用 BGE-Large-ZH 本地部署。该模型在 C-MTEB 中文榜长期排名靠前，1.3GB 模型大小可在 2 核 4G 服务器上运行，LangChain 原生支持集成。

**Token 消耗**：每次问答约 **2000-5000 Token**

### 4.2 场景 2：ArkTS 代码生成

**输入**：自然语言需求（如"写一个鸿蒙待办清单 App，支持添加、删除、本地存储"）

**处理流程**：
1. 用 MiMo 生成代码框架（.ets 文件）
2. 调用 MiMo 生成配套 XML 布局（如需要）
3. 用 MiMo 生成单元测试代码
4. 返回完整可运行的 DevEco Studio 项目

**Token 消耗**：每次生成约 **5000-10000 Token**

### 4.3 场景 3：报错诊断

**输入**：编译错误信息（如"Cannot find module '@ohos/net'..."）

**处理流程**：
1. 用 MiMo 分析错误类型和可能原因
2. 检索知识库中的相似案例
3. 生成分步修复方案（含代码示例）
4. 返回 Markdown 格式的诊断报告

**Token 消耗**：每次诊断约 **3000-6000 Token**

### 4.4 场景 4：Android → ArkTS 代码迁移

**输入**：Android Java/Kotlin 代码

**处理流程**：
1. 用 MiMo 理解原始代码逻辑
2. 生成等价的 ArkTS 实现
3. 标注迁移注意事项（如 API 差异）
4. 返回对比代码（Before/After）

**Token 消耗**：每 100 行代码约 **8000-15000 Token**

### 4.5 场景 5：CSDN 文章自动生成

**输入**：问答对 + 代码案例

**处理流程**：
1. 用 MiMo 将技术内容整理成文章大纲
2. 生成完整 Markdown 文章（含代码块、配图提示词）
3. 调用 CSDN API 自动发布到草稿箱
4. 返回文章链接

**Token 消耗**：每篇文章约 **4000-8000 Token**

### 4.6 场景 6：Agent 工具调用

**输入**：用户复合需求（如"帮我查一下 Button 组件的用法，然后写个示例，再看看有没有编译问题"）

**处理流程**：
1. MiMo 解析用户意图，拆解为子任务序列（检索文档 → 生成代码 → 诊断编译）
2. ReAct 循环：思考(Thought) → 行动(Action，调用 Tool) → 观察(Observation) → 下一步
3. 每个 Tool 调用独立记录：Tool 名称、输入参数、返回结果
4. 最终汇总所有结果，流式返回完整答案

**Token 消耗**：每次 Agent 调用约 **5000-10000 Token**（含多轮 Tool 调用上下文）

### 4.7 场景 7：多轮对话与记忆管理

**输入**：用户在对话中逐步细化需求

**处理流程**：
1. 对话历史窗口管理：根据 Token 计数自动滑动，保持上下文在模型限制内
2. 长对话摘要压缩：当历史超过阈值，用 MiMo 将早期对话总结为 200 字摘要
3. 用户偏好提取：从对话中提取用户习惯（如常用组件、代码风格），写入偏好库
4. 后续对话自动注入偏好上下文

**Token 消耗**：每次对话额外消耗约 **1000-2000 Token**（摘要 + 偏好注入）

### 4.8 场景 8：高级 RAG（Query Rewriting + Rerank）

**输入**：用户模糊问题（如"那个蓝色按钮怎么调"）

**处理流程**：
1. Query Rewriting：用 MiMo 将口语化问题改写为精确检索 Query（"ArkUI Button 组件 backgroundColor 属性设置方法"）
2. 混合检索：BM25（关键词匹配）+ 稠密向量（语义匹配）→ 各取 Top-10 → 合并去重
3. BGE-Reranker 对候选文档重排序，取 Top-5
4. 进入标准 RAG 问答流程

**Token 消耗**：每次问答额外增加约 **500-1000 Token**（Query Rewriting）

### 4.9 场景 9：评测数据自动生成

**输入**：知识库文档

**处理流程**：
1. 从文档中自动抽取关键知识点
2. 用 MiMo 为每个知识点生成问答对（Question-Answer）
3. 人工抽检 + 修正，构建 Ground Truth 评测集
4. 入库评测数据集，用于 RAGAS 自动化回归

**Token 消耗**：每篇文档生成约 **2000-5000 Token**

### 4.10 场景 10：Graph RAG 实体抽取

**输入**：鸿蒙 API 文档

**处理流程**：
1. 用 MiMo 从文档中抽取实体（组件、API、属性、事件）及关系（继承、参数类型、返回值）
2. 构建三元组 (Entity-Relation-Entity) 写入 Neo4j
3. 检索时沿图遍历关联节点，增强上下文
4. 例如：检索 Button → 图遍历发现继承自 Component → 同时返回 Component 通用属性

**Token 消耗**：每篇文档约 **3000-6000 Token**（实体关系抽取）

### 4.11 场景 11：多模态 UI 识别

**输入**：鸿蒙 App 界面截图

**处理流程**：
1. 截图上传，调用 Vision Model 识别 UI 布局（组件类型、位置、样式）
2. 输出结构化 UI 描述（JSON 格式：组件树 + 属性）
3. 用 MiMo 将 UI 描述转换为 ArkUI 代码
4. 编译错误截图则先 OCR 提取文本，再进入报错诊断流程

**Token 消耗**：每次识别 + 代码生成约 **8000-15000 Token**（含图片 Token）

### 4.12 场景 12：SFT 数据集构造

**输入**：用户问答记录 + 赞/踩反馈

**处理流程**：
1. 筛选高赞问答对，用 MiMo 清洗和规范化
2. 生成 instruction-input-output 三元组格式
3. 数据增强：用 MiMo 对同一知识点生成多种问法变体
4. 输出标准 Alpaca 格式 JSONL 数据集，供 LoRA 微调使用

**Token 消耗**：每 100 条数据约 **10000-20000 Token**

## 六、风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| MiMo Token 申请失败 | 中 | 高 | LLM 抽象层已支持 DeepSeek/OpenAI 兼容接口降级 |
| Embedding 模型效果不佳 | 低 | 中 | BGE-Large-ZH 在 C-MTEB 中文榜表现优秀；备选 BGE-M3（多语言+混合检索）或 GTE-Qwen2 |
| 华为文档站 SPA 渲染困难 | 高 | 高 | 使用 Playwright 无头浏览器渲染；备选 OpenHarmony 仓库 Markdown 文档（无需渲染） |
| 华为文档爬取被封禁 | 中 | 中 | 遵守 robots.txt、降低爬取频率、缓存已爬取内容、备用 OpenHarmony 仓库 |
| 文档分块质量不佳 | 中 | 中 | 按语义标题切分而非固定字数、保留代码块完整性、人工抽检 chunk 质量 |
| 向量数据库成本过高 | 低 | 中 | 使用免费版 Milvus、或本地部署 Qdrant |
| Agent 工具调用准确性不足 | 中 | 高 | ReAct 循环限制最大步数（≤5 步）、每步校验 Tool 输出合法性、人工标注 Bad Case 持续优化 Prompt |
| 语义缓存命中率低 | 中 | 低 | 调整相似度阈值（0.90-0.98 区间实验）、分场景独立设置阈值、监控命中率并动态调优 |
| Vision Model 识别准确率低 | 高 | 中 | 先用简单 UI（低组件数）验证流程、输出结构化 JSON 而非直接代码、保留人工修正入口 |
| Prompt Injection 安全事件 | 低 | 高 | 输入层关键词过滤 + 输出层代码沙箱校验、敏感操作二次确认、定期更新注入规则库 |
| 开源模型微调效果不达预期 | 中 | 中 | 先用 100 条小数据集验证 Loss 收敛、对比 Full Fine-tune vs LoRA、必要时使用更大参数量的基座模型 |
| 服务成本超预算 | 中 | 中 | 语义缓存降低重复调用、设置日 Token 消耗上限、非高峰时段降级使用便宜模型 |
| 项目进度延期 | 中 | 低 | M0-M2 为核心 MVP 必须交付，M3-M8 按优先级和实际资源排入后续迭代 |
